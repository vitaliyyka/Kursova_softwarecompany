АНАЛІЗ ПРЕДМЕТНОЇ ОБЛАСТІ
Аналіз вимог та інформаційних потреб
Вимоги користувачів до бази даних (БД) для інформаційної системи компанії з розробки програмного забезпечення можуть виглядати так:
1. Запит на перегляд списку проектів:
Інтенсивність: Висока
   Об'єм даних: Малий до Середнього
   Опис: Користувачі хочуть бачити список доступних проектів разом з їхніми назвами, статусами та відповідними командами розробників.
2. Запит на пошук проектів за ключовими словами або параметрами:
Інтенсивність: Середня
   Об'єм даних: Малий до Середнього
   Опис: Користувачі хочуть мати можливість шукати проекти за ключовими словами, описом, тегами або параметрами (наприклад, технологія, мова програмування).
3. Запит на перегляд інформації про конкретний проект:
   Інтенсивність: Висока
   Об'єм даних: Малий до Середнього
   Опис: Користувачі хочуть бачити детальну інформацію про обраний проект, включаючи опис, статус, склад команди, графік та інші характеристики.
4. Запит на створення нового проекту:
   Інтенсивність: Середня
   Об'єм даних: Малий до Середнього
   Опис: Користувачі хочуть мати можливість створювати нові проекти, вказуючи назву, опис, дату початку, термін виконання та інші параметри.
5. Запит на призначення розробників до проекту:
   Інтенсивність: Середня
   Об'єм даних: Малий до Середнього
   Опис: Користувачі хочуть мати можливість додавати та змінювати склад команди для кожного проекту, призначаючи розробників до відповідних завдань та ролей.
6. Запит на створення та оновлення завдань у проекті:
   Інтенсивність: Висока
   Об'єм даних: Малий до Середнього
   Опис: Користувачі хочуть мати можливість створювати нові завдання, описуючи їх деталі, пріоритети, дедлайни та призначати відповідних розробників.
7. Запит на перегляд статусу та звітності про проекти:
   Інтенсивність: Середня
   Об'єм даних: Середній до Великого
   Опис: Користувачі хочуть бачити статус виконання кожного проекту, а також мати можливість генерувати звіти та статистику згідно з різними параметрами.
8. Запит на управління користувачами та доступом до проектів:
   Інтенсивність: Висока
   Об'єм даних: Малий до Середнього
   Опис: Адміністратори хочуть мати можливість керувати правами доступу користувачів до різних проектів, а також реєструвати та автентифікувати нових користувачів.
9. Запит на перегляд та аналіз даних про ресурси та фінанси:
   Інтенсивність: Середня
   Об'єм даних: Великий
   Опис: Менеджери та фінансові аналітики хочуть мати можливість аналізувати дані про ресурси, фінансові витрати та прибуток, розподіл бюджету між проектами та виконавцями.

Виявлення інформаційних об’єктів і зв’язків між ними
1. Клієнт (Client):
 ID клієнта (Client_ID) - первинний ключ
 Ім'я (First Name)
 Прізвище (Last Name)
 Адреса (Address)
 Номер телефону (Phone Number)
 Електронна пошта (Email)
2. Проект (Project):
 ID проекту (Project_ID) - первинний ключ
 Назва проекту (Project Name)
 Опис проекту (Project Description)
 Дата початку (Start Date)
 Дата завершення (End Date)
 Бюджет (Budget)
3. Співробітник (Employee):
 ID співробітника (Employee_ID) - первинний ключ
 Ім'я (First Name)
 Прізвище (Last Name)
 Електронна пошта (Email)
 Номер телефону (Phone Number)
 Дата прийняття на роботу (Hire Date)
 Посада (Job Title)
 ID відділу (Department_ID) - зовнішній ключ, пов'язаний з ID відділу
4. Завдання (Tasks):
 ID завдання (Task_ID) - первинний ключ
 Назва завдання (Task Name)
 Опис завдання (Task Description)
 Статус (Status)
 Дата початку (Start Date)
 Дата завершення (End Date)
 ID проекту (Project_ID) - зовнішній ключ, пов'язаний з ID проекту
 ID відповідального (Assigned To) - зовнішній ключ, пов'язаний з ID співробітника
5. Статус завдання(Task status):
 ID статусу (Task_Status_ID) - первинний ключ
 Назва статусу (Status Name)

Зв'язки між таблицями в базі даних можна розписати наступним чином:

1. Зв'язок між таблицею "Клієнт" і "Проект":
Клієнт може мати багато проектів (One-to-Many зв'язок).
В таблиці "Проект" є зовнішній ключ "Client_ID", який посилається на ID клієнта в таблиці "Клієнт".

2. Зв'язок між таблицею "Проект" і "Співробітник":
Проект може мати багато співробітників, а співробітник може брати участь в багатьох проектах (Many-to-Many зв'язок).
Для реалізації такого зв'язку створюється додаткова таблиця, наприклад, "Участь у проекті", яка містить зовнішні ключі "Project_ID" і "Employee_ID", які посилаються на відповідні ID у таблицях "Проект" і "Співробітник".

4. Зв'язок між таблицею "Проект" і "Завдання":
Проект може мати багато завдань (One-to-Many зв'язок).
В таблиці "Завдання" є зовнішній ключ "Project_ID", який посилається на ID проекту в таблиці "Проект".

5. Зв'язок між таблицею "Завдання" і "Співробітник":
Завдання може бути призначено одному співробітнику, але один співробітник може мати багато завдань (One-to-Many зв'язок).
В таблиці "Завдання" є зовнішній ключ "AssignedTo", який посилається на ID співробітника в таблиці "Співробітник".
